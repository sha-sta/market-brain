# MarketBrain — what is novel, what already exists, who it is for

## TL;DR — honest verdict

MarketBrain is a **novel synthesis, not a novel mechanism.** Every individual ingredient — LLM extraction of a typed entity/edge graph, self-updating agent memory with ADD/UPDATE/DELETE reconciliation, reference-guarded deletion, verbatim-substring evidence grounding, deterministic floor-enforcement on an LLM verdict, and a non-advisory adversarial "devil's advocate" critic — has clear, citable prior art. What I could not find in any single shipped project is the **specific assembly**: a *personal, finance-domain* knowledge graph whose daily cron BOTH grows the graph from live market/news feeds AND decays it via tiered, **reference-guarded hard-delete** (reclaiming row + embedding while refusing to delete live-thesis evidence), wired to a verbatim-evidence-gated adversarial critic that can only demote its verdict, all on a single Supabase free tier with cross-layer build-failing invariants. The defensible claim is "this combination, this enforcement rigor, at this cost" — **not** "first to do X" for any single X. The closest single competitors are Cognee (durable self-updating personal KG with a "what to forget" story) on the memory axis and the open-source `devilsadvocate` repo on the non-advisory-critic axis; both are cited honestly below.

## Per-differentiator verdicts

**1. Grow-AND-decay personal finance KG on a cron — `partial-overlap`.**
Each axis has prior art but no single OSS project bundles them. Nearest: [gbrain](https://github.com/garrytan/gbrain) (personal Postgres+pgvector self-wiring KG with cron enrichment + contradiction/staleness detection) — but it is general-purpose, human-git-authored, and deliberately **never deletes** (git deletes become soft-deletes; staleness is only flagged). Runner-up [FinDKG](https://xiaohui-victor-li.github.io/FinDKG/) proves finance + dynamic + scheduled exists, but as an append-only research dataset, not a personal grow-and-decay tool.

**2. Reference-guarded hard-delete for storage reclamation — `partial-overlap`.**
The "don't delete if still referenced" half is textbook (SQL `ON DELETE RESTRICT`, GC reference-counting); the "age-based expiry that reclaims space only when nothing live refers to the node" half is essentially [US Patent 10725982 "Knowledge graph node expiration"](https://image-ppubs.uspto.gov/dirsearch-public/print/downloadPdf/10725982) and the SBU dependency-graph reference counter in ["Agentic Unlearning"](https://arxiv.org/html/2602.17692v1). MarketBrain's uncommon twist is a **domain-semantic** guard (live-thesis evidence + active tracked entity) rather than a generic FK/refcount — and it is the *opposite* of the agent-memory mainstream ([Graphiti invalidates-not-deletes](https://neo4j.com/blog/developer/graphiti-knowledge-graph-memory/); [Mem0 hard-deletes contradicted memories with no guard](https://docs.mem0.ai/core-concepts/memory-operations/update); Letta refuses to auto-delete and [accretes duplicates](https://github.com/letta-ai/letta/issues/3116)).

**3. Self-updating memory that reconciles facts (ADD/UPDATE/DELETE) — `already-exists`.**
[Mem0](https://docs.mem0.ai/core-concepts/memory-operations/update) productizes exactly this extract→reconcile loop, on the same Postgres+pgvector stack, with a paper and benchmarks ([arXiv 2504.19413](https://arxiv.org/html/2504.19413v1)); its V3 `linked_memory_ids` supersede chain parallels MarketBrain's `superseded_by`. Do **not** claim self-updating memory as a differentiator.

**4. Fact reconciliation riding the same extraction LLM call (zero extra cost) — `partial-overlap`.**
Fact correction itself is commoditized (Mem0, [Graphiti edge invalidation](https://github.com/getzep/graphiti)). The narrow delta is the **cost packaging**: corrections to permanent nodes ride the extraction call's own output schema (`reconcile.ts:13` — "Zero added LLM cost"), whereas [Mem0's reconcile step](https://mem0.ai/blog/mem0-the-token-efficient-memory-algorithm) and Graphiti's invalidation both spend a *separate* LLM pass. Verbatim-gated, ≥0.85 auto-apply / 0.6–0.85 queued, identity fields protected (rename → `former_name`/`aliases`, never overwrite). A real but narrow engineering/cost optimization, not a research contribution.

**5. Verbatim-substring evidence gate on an edge's `assertable` flag — `partial-overlap`.**
Verbatim-or-invalid quoting is established ([Deterministic Quoting](https://mattyyeung.github.io/deterministic-quoting), [Registry-Bound LLM Pipeline arXiv 2606.00994](https://arxiv.org/pdf/2606.00994) pairs verbatim substring verification *with* confidence stratification — the same combo). The thin delta is fusing whitelist + confidence ≥ 0.8 + NFC verbatim substring into one hard `assertable` boolean on a typed persistent graph edge; no exact match found for that packaging.

**6. Deterministic, demote-only floor on an LLM verdict — `partial-overlap` (the pattern `already-exists`).**
"Code, not the LLM, computes the final verdict" is the canonical design of [RAGAS](https://docs.ragas.io/en/stable/concepts/metrics/available_metrics/faithfulness/), [FActScore](https://arxiv.org/abs/2305.14251), and especially [DeepEval's DAG metric](https://deepeval.com/docs/metrics-dag) ("the score comes from the deterministic path taken, not the LLM picking a number"), which is functionally `enforceFloor()`. The distinctive twist is **asymmetric, conservative-only** flooring (can demote toward verified evidence, never inflate) on a *user-facing thesis verdict* in production, contrasted with the symmetric [Judge Pattern clamp](https://blog.pytoshka.me/en/post/the-judge-pattern-llm-verdicts/). The field's usual anti-sycophancy answer is *more* LLMs (juries/meta-judges); MarketBrain's is *less* LLM authority.

**7. Non-advisory adversarial critic on the user's own thesis — `already-exists` (combination `partial-overlap`).**
This is **not novel.** The open-source [devilsadvocate](https://github.com/unicodeveloper/devilsadvocate) makes no buy/sell/hold/price-target calls, runs a multi-agent critic (Bull/Bear/House-View/Synthesizer) on the user's own memo, and floors to an Approved/Changes-Requested/Rejected verdict via a deterministic hard rule — over SQLite. The commercial [LinqAlpha "Devil's Advocate"](https://aws.amazon.com/blogs/machine-learning/how-linqalpha-assesses-investment-theses-using-devils-advocate-on-amazon-bedrock/) pressure-tests your own thesis with neutral, citation-grounded output. MarketBrain must **cite** these, not claim it invented "attack your own thesis." Its delta: the critic rides on top of a living decaying graph and a verbatim-substring evidence gate, neither of which those two have. By contrast, [Danelfin](https://danelfin.com/how-it-works) and [Kavout](https://www.kavout.com/k-score/) emit literal Buy/Sell signals and [Fintool](https://fintool.com/products/research) generates ratings/price targets/DCFs — the philosophical opposites, useful for honest contrast.

**8. Cross-layer build-failing invariants (SQL literal == TS constant) — `partial-overlap`.**
The "two schema representations must agree or the build fails" category is commoditized ([Supabase codegen:check + git-diff guards](https://supabase.com/docs/guides/deployment/ci/generating-types), Atlas/Liquibase drift detectors). What is uncommon — but a trivial composition of `readFileSync` + regex + `expect().toEqual()` — is keeping *three hand-authored* copies (SQL generated-column literal, TS `STRONG_RELATIONS`, `isAssertable()`) plus double-sourced decay windows, and guarding them by regex-parsing the raw `.sql` migration text in a unit test (`tests/unit/relations.test.ts`, `lifecycle.test.ts`). Frame as engineering discipline, not a novel mechanism.

## Why not Graphiti/Zep (or an off-the-shelf KG framework)?

[Graphiti/Zep](https://github.com/getzep/graphiti) is the most-cited "temporal knowledge-graph memory" framework, and on retrieval (LLM-free hybrid vector + BM25 + graph traversal, RRF/cross-encoder rerank, built-in dedup) it is genuinely better than what MarketBrain hand-rolled — that part should be *measured against*, not assumed inferior. But it is architecturally incompatible with MarketBrain's two founding premises:

1. **It requires a graph database.** Neo4j 5.26+, FalkorDB, Neptune+OpenSearch, or the deprecated Kuzu — there is **no Postgres/pgvector driver at all**. MarketBrain's entire premise is one Supabase free tier (Postgres + pgvector, RLS, one cron). Adopting Graphiti means standing up and paying for a persistent graph-DB server; the [Community Edition was discontinued in April 2025](https://blog.getzep.com/announcing-a-new-direction-for-zeps-open-source-strategy/) and Zep Cloud is credit-metered (free ≈ 2.5 MB, Flex $104/mo).

2. **It is invalidate-only by design philosophy.** Graphiti is bi-temporal and **never deletes rows** — its stated goal is ["historical completeness rather than storage optimization"](https://blog.getzep.com/beyond-static-knowledge-graphs/) ([arXiv 2501.13956](https://arxiv.org/html/2501.13956v1)). That is the *exact opposite* of MarketBrain's reference-guarded hard-delete, which exists specifically to reclaim row + embedding under a fixed ~500 MB budget.

It is also domain-agnostic: no finance tooling, no non-advisory invariant, no adversarial critic, no verbatim-evidence gating of edges. The same logic rules out the document-corpus builders: [GraphRAG](https://microsoft.github.io/graphrag/index/architecture/) is an append-only batch artifact, [LlamaIndex PropertyGraphIndex](https://developers.llamaindex.ai/python/framework/module_guides/indexing/lpg_index_guide/) gives durable upsert/delete *primitives* but no lifecycle policy, and [Cognee](https://www.cognee.ai/blog/fundamentals/how-cognee-builds-ai-memory) — the closest on the "what to forget" axis — deletes only at [dataset granularity or by usage reweighting](https://docs.cognee.ai/python-api/prune), not per-node reference-guarded prune. **Custom Postgres-native build is defensible**; the honest caveat is that Graphiti's retrieval+dedup is the part MarketBrain reimplemented and should be benchmarked against.

## Who this is for

- **The primary user is one person** (this is a Father's Day gift): an individual investor who tracks a watchlist of names/industries and wants a research brain that maintains itself — growing from a daily feed, forgetting what no longer matters, correcting facts when sources change, and producing a strict daily brief.
- **It is for readers who want evidence, not advice.** The non-advisory invariant + verbatim-evidence gate + adversarial critic exist for someone who wants their *own* thesis stress-tested against source-grounded facts and is willing to make the decision themselves. It is explicitly **not** for anyone who wants a buy/sell signal (use Danelfin/Kavout) or analyst ratings/DCFs (use Fintool/Fiscal.ai).
- **As an engineering artifact**, it is for reviewers evaluating disciplined system design under hard constraints: cost firewall, build-failing cross-layer invariants, deterministic safety backstops, and SSRF-hardened untrusted-web ingestion.

## Portfolio strength — the skeptic's view

**Above a typical impressive OSS LLM app.** The median "impressive" LLM project is a RAG chatbot or an agent wrapper with a clean demo and shallow operational depth. MarketBrain is meaningfully stronger on the axes that distinguish senior engineering from prompt-gluing: (a) it ships a *correctness regime* — cross-layer build-failing invariants that fail the build on SQL↔TS drift, a verbatim-substring evidence gate, and a deterministic demote-only verdict floor — i.e. it treats LLM output as untrusted and backstops it with code; (b) it makes a real, defended cost/architecture tradeoff (Postgres-only, one cron, ~500 MB, reference-guarded hard-delete) instead of reaching for a graph DB; (c) it has a coherent safety posture (non-advisory invariant, SSRF guards, cost firewall on auto-discovered candidates) and a genuine test suite (144 unit / 56 integration against a real DB / e2e).

**The strongest counterargument:** *none of the headline mechanisms is novel, and the author can fairly be accused of reinventing wheels that mature libraries already turn better.* Self-updating fact-reconciling memory is Mem0 (productized, benchmarked, same stack). The deterministic verdict floor is DeepEval's DAG metric. Verbatim grounding is Deterministic Quoting. The non-advisory adversarial critic is the `devilsadvocate` repo almost feature-for-feature. Hybrid retrieval + dedup — the one part that is genuinely hard — is exactly where Graphiti is *better* and MarketBrain's hand-rolled version is unmeasured. A skeptical reviewer would say: the engineering rigor is real, but the system is a competent *recombination and re-implementation* of existing ideas in a Postgres-only box, and a stronger portfolio move would have been to either (i) benchmark the reimplemented retrieval against Graphiti to prove it isn't worse, or (ii) build on an existing memory framework and spend the saved effort on the one defensible novelty — the domain-semantic reference-guarded hard-delete — and evaluate it rigorously. The rebuttal: integration *is* the skill on display, the constraints are real, and the combination genuinely does not exist as a shipped product — but the honest verdict is "strong synthesis with disciplined execution," not "novel research."
